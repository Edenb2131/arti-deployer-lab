# Keycloak overlay

Spins up Keycloak as an OIDC identity provider and wires Artifactory 1
to trust it in two ways:

1. **Access OIDC token exchange** (AF 7.100+ API) — for CI/CD scripts
   that already have a Keycloak token and want to swap it for an AF token.
2. **OAuth SSO login button** — a "Sign in with Keycloak" button on the
   AF UI login page.

## What runs

- **Container:** `arti-keycloak` (image: `quay.io/keycloak/keycloak:25.0`)
- **Mode:** `start-dev` (embedded H2; not for production)
- **Port:** `${KEYCLOAK_PORT}` → 8080
- **Realm:** `jfrog` (pre-imported from `config/keycloak/realm.json` rendered
  from `realm.json.tmpl`)
- **Admin:** `admin` / `${KEYCLOAK_ADMIN_PASSWORD}`

## Realm contents

| Item | Value |
|---|---|
| Client ID | `artifactory` |
| Client Secret | `${KEYCLOAK_CLIENT_SECRET}` (auto-generated by `init`) |
| Test user | `testuser` / `${LDAP_USER_PASSWORD}` |
| Admin user | `kcadmin` / `${LDAP_USER_PASSWORD}` |
| Roles | `developer`, `admin-role` |
| Redirects | `localhost:8081/*`, `localhost:8082/*`, `localhost:8080/*`, `https://localhost:8443/*`, `http://artifactory1:8082/*` |

## How AF gets wired (3 steps in `configure-keycloak.sh`)

### Step 1 — Get a bearer token

```
POST /access/api/v1/tokens
Auth: Basic admin:<AF_ADMIN_PASSWORD>
{"username":"admin","scope":"applied-permissions/admin","expires_in":3600}
```

AF 7.100+ requires Bearer auth on `/access/api/v1/*`. Basic auth no longer
works on those endpoints — old guides may be misleading on this.

### Step 2 — Register the OIDC provider

```
POST /access/api/v1/oidc
Auth: Bearer <token from step 1>
{
  "name": "keycloak",
  "issuer_url": "http://localhost:8090/realms/jfrog",
  "audience": "artifactory",
  "description": "...",
  "provider_type": "generic",          ← lowercase! Old "GENERIC" returns 400
  "enable_permissive_configuration": false,
  "use_default_proxy": false
}
```

Notes vs the pre-7.100 API:
- `provider_type` is **lowercase** (`generic`, `github`, `azure`, ...).
- `client_id` / `client_secret` are NOT in this payload — token-exchange
  flow doesn't need them; the issuer's JWT signature is what AF validates.

### Step 3 — Identity mapping

```
POST /access/api/v1/oidc/keycloak/identity_mappings
Auth: Bearer <token>
{
  "name": "keycloak-testuser",
  "priority": 1,
  "claims": {"preferred_username": "testuser"},
  "token_spec": {
    "username": "testuser",
    "scope": "applied-permissions/groups:readers",
    "expires_in": 7200
  }
}
```

When a Keycloak token with `preferred_username: testuser` is exchanged,
AF mints a token as user `testuser` with `readers` group permissions.

### Step 4 (separate) — OAuth SSO UI button

This is the **older** API (`/artifactory/api/system/configuration` PATCH
with `security.oauthSettings`). It's distinct from the Access OIDC flow
above. As of 7.100+ they still coexist — Access OIDC is for tokens, OAuth
SSO is for the user login UI.

## Token-exchange test (CI/CD-style)

```bash
# Get a Keycloak token (using direct grant in dev mode):
KC_TOKEN=$(curl -s "http://localhost:8090/realms/jfrog/protocol/openid-connect/token" \
  -d "client_id=artifactory" \
  -d "client_secret=$(grep ^KEYCLOAK_CLIENT_SECRET .env | cut -d= -f2)" \
  -d "username=testuser" \
  -d "password=$(grep ^LDAP_USER_PASSWORD .env | cut -d= -f2)" \
  -d "grant_type=password" | jq -r .access_token)

# Exchange it for an AF token:
curl -s "http://localhost:8082/access/api/v1/oidc/token" \
  -H 'Content-Type: application/json' \
  -d "{
    \"subject_token\": \"${KC_TOKEN}\",
    \"subject_token_type\": \"urn:ietf:params:oauth:token-type:id_token\",
    \"provider_name\": \"keycloak\"
  }" | jq
```

You should get back an AF access token in the response.

## UI login flow

Browser → `http://localhost:8081/ui/` → "Sign in with keycloak" button →
redirects to Keycloak login → log in as `testuser` → redirected back
authenticated.

## Internal vs external URLs (gotcha)

- `issuer_url` and the auth-redirect URL must be reachable from your
  **browser** → `http://localhost:${KEYCLOAK_PORT}/...`
- The token URL and userinfo URL are called by AF **from inside its
  container** → `http://keycloak:8080/...`

The pre-imported realm has both forms in `redirectUris`.

## Troubleshooting

**`POST /access/api/v1/oidc` returns 401** — your AF version is older than
7.100, or the bearer token expired. Re-run `configure-keycloak.sh`.

**Returns 400 with "provider_type"** — check the payload uses `"generic"`
not `"GENERIC"`. Case matters in 7.100+.

**Invalid redirect_uri** — your AF URL isn't in the realm's allow-list.
Add it to `config/keycloak/realm.json.tmpl` redirectUris and run
`./arti-deployer reset && up` so Keycloak re-imports the realm.

**Token exchange returns "no mapping matched"** — the OIDC token's claims
don't match any identity mapping. Check the Keycloak token contents at
[jwt.io](https://jwt.io) and adjust the `claims` filter in step 3.
